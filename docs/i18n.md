# Internationalisation (i18n)

SoroStream uses a lightweight, file-based i18n system located in `src/lib/i18n.ts` with message files in `messages/`.

---

## Key Naming Convention

Keys follow a **`namespace.element`** pattern, where each namespace corresponds to a top-level object in the message file.

```
<namespace>.<element>
```

Examples used in the codebase:

| Key | Meaning |
|-----|---------|
| `dashboard.title` | Page heading on the Dashboard |
| `wallet.connect` | "Connect {wallet}" button label |
| `stream_detail.not_found_title` | 404 heading on the stream detail page |
| `settings.contact_count` | "{count}/{max} contacts" counter |

**Convention rules:**
- Use `snake_case` for multi-word elements (`no_streams`, `hero_title`).
- Namespace names match the page or feature they belong to (`home`, `dashboard`, `stream_new`, `stream_detail`, `wallet`, `settings`).
- Interpolation placeholders use `{camelCase}` (`{wallet}`, `{id}`, `{count}`).

---

## Currently Supported Locales

| Locale | File | Status |
|--------|------|--------|
| `en` (English) | `messages/en.json` | Default / complete |

The hook always loads English (`en`) at runtime. Adding a new locale is described below.

---

## Adding a New Language

1. **Create a message file** at `messages/<locale>.json` (e.g. `messages/fr.json`) with the same top-level structure as `en.json`:

   ```json
   {
     "home": {
       "hero_title": "Envoyez de l'argent comme des donnĂ©es",
       ...
     },
     "dashboard": { ... },
     ...
   }
   ```

2. **Update `src/lib/i18n.ts`** to load the correct locale. The current implementation always imports `en`; extend it with a locale lookup:

   ```ts
   import en from "../../messages/en.json";
   import fr from "../../messages/fr.json";

   const locales = { en, fr } as const;
   type LocaleKey = keyof typeof locales;

   const activeLocale: LocaleKey =
     (typeof navigator !== "undefined" && navigator.language.slice(0, 2) as LocaleKey in locales)
       ? (navigator.language.slice(0, 2) as LocaleKey)
       : "en";

   const messages = locales[activeLocale];
   ```

3. **Keep all keys in sync.** Every key present in `en.json` must also exist in the new locale file. Missing keys fall back to the raw key string.

## Using `useTranslations` in a Component

```tsx
import { useTranslations } from "@/src/lib/i18n";

export default function MyComponent() {
  const t = useTranslations("dashboard"); // pass the namespace

  return <h1>{t("title")}</h1>;          // â†’ "Dashboard"
}
```

### With interpolation

```tsx
const t = useTranslations("wallet");

// message: "Connect {wallet}"
t("connect", { wallet: "Freighter" }); // â†’ "Connect Freighter"
```

### With multiple placeholders

```tsx
const t = useTranslations("settings");

// message: "{count}/{max} contacts"
t("contact_count", { count: "3", max: "10" }); // â†’ "3/10 contacts"
```

All interpolation values must be `string`. Convert numbers before passing them: `String(n)`.

## Handling Pluralisation

The current hook does not have built-in plural support. The recommended pattern is to use separate keys:

```json
"stream_count_one": "1 stream",
"stream_count_other": "{count} streams"
```

Then choose the key in the component:

```tsx
const key = count === 1 ? "stream_count_one" : "stream_count_other";
t(key, { count: String(count) });
```

For richer plural rules (e.g. Slavic languages), replace the hook body with a call to the [Intl.PluralRules](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Intl/PluralRules) API.

---

## Handling Interpolation Edge Cases

- **Missing placeholder**: if a `{key}` in the message string has no matching entry in `vars`, it is left as-is in the output.
- **Extra vars**: unused entries in `vars` are silently ignored.
- **Undefined key**: if a translation key does not exist in the message file, the raw key string is returned (e.g. `"dashboard.missing_key"`).
